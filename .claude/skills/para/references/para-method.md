# PARA Method — Methodology Reference

## Core Principle: Actionability, Not Topic

Traditional filing organizes by subject ("Java", "Infrastructure"). PARA organizes by **when and
how you'll use the information**:

| Category | Question | Timeframe |
|---|---|---|
| **Project** | What am I working on right now? | Days to weeks |
| **Area** | What am I responsible for? | Ongoing |
| **Resource** | What might I need someday? | Someday |
| **Archive** | What have I already finished? | Historical |

**Example:** A Kubernetes folder is a Project if you're actively deploying right now, an Area if
you're responsible for a running cluster, or a Resource if you're just learning for reference.

---

## The Four Categories

### Project

**Definition:** A series of tasks connected to a specific goal with a deadline.

**Characteristics:**
- Has a clear outcome that can be "done"
- Has a timeline (even if approximate)
- Can be archived once complete
- All required material is gathered here

**Language signals:** "complete", "deliver", "launch", "finish", "ship", "build", "implement"

### Area

**Definition:** A scope of responsibility that requires ongoing maintenance, with no end date.

**Characteristics:**
- Never "done"
- Represents a role or responsibility
- Has a standard to maintain
- The source new projects are born from

**Language signals:** "maintain", "manage", "oversee", "monitor", "keep up"

**Critical distinction — Project vs Area:**
> You don't "finish" the responsibility of teaching a course — you carry it out (Area).
> You *can* finish "Build the spring-semester course module" (a Project born from that Area).

### Resource

**Definition:** A topic of interest, not tied to an active project or responsibility.

**Characteristics:**
- No commitment to act
- Reference material for future needs
- Can feed into a project when needed

**Language signals:** "interesting", "might need", "reference", "exploration", "learning"

**Warning:** Resource easily becomes a dumping ground. Review periodically and remove what's never accessed.

### Archive

**Definition:** An inactive item from the three categories above.

**Characteristics:**
- "Cold storage" — not deleted, just hidden from active view
- Still searchable when needed
- Preserves history and context

**When to archive:**
- A project is finished or cancelled
- An area of responsibility ends (role change)
- A resource is no longer relevant

---

## Lifecycle

```
Resource ──────────────────────────────────► Archive
             ↓ (an active commitment appears)
           Area ──────────────────────────► Archive
             ↓ (a specific goal + deadline appears)
           Project ────────────────────────► Archive
```

Items move between categories as status changes — this is not a permanent assignment.

---

## Just-in-Time Organization

**Do:**
- Create a folder only when there's real content to put in it
- Let structure emerge from actual work
- Move items between categories as status changes
- Archive aggressively — the active view should stay clean

**Avoid:**
- Pre-creating empty folders as placeholders
- Spending more time organizing than working
- Treating categories as permanent

---

## Common Mistakes

| Mistake | Symptom | Fix |
|---|---|---|
| Organizing by topic, not actionability | Can't find something when you need it | Ask "when will I use this?" not "what is this about?" |
| Too many projects | Overwhelm, nothing gets finished | Be honest: how many projects can you run in parallel? Archive inactive ones |
| Areas never reviewed | Standards slip, reactive instead of proactive | Areas need regular attention, not just projects |
| Resource becomes a dumping ground | Folder grows unbounded, never found again | Delete resources not accessed in > 6 months |
| Never archiving | Active folders full of old work | Archive is searchable, not deleted |
